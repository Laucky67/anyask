# 快捷提问窗口聚焦切换与重置策略设计

## 背景

当前快捷提问全局快捷键的行为是：

1. 快捷提问窗口不存在或隐藏时，按快捷键呼出窗口。
2. 快捷提问窗口已显示时，按快捷键隐藏窗口。

当快捷提问窗口被其它窗口覆盖但仍处于显示状态时，用户第一次按快捷键会隐藏它，第二次按快捷键才重新显示，实际需要按两次才能看见窗口。

同时，快捷提问窗口承载 AI 网页，长期隐藏但保留 WebView 会带来性能和内存开销。需要新增基础配置，让用户控制隐藏后何时注销快捷提问窗口。用户文案统一使用“关闭”，实现上对应隐藏窗口。

## 目标

- 快捷提问窗口已显示但未聚焦时，按快捷键应把窗口移至最上方，而不是隐藏。
- 快捷提问窗口已显示且聚焦时，按快捷键隐藏窗口。
- 基础配置新增“快捷提问重置为新对话”选项。
- 达到配置时间后注销隐藏状态下的快捷提问窗口，下次呼出创建新的快捷提问窗口。
- “重新打开时”表示隐藏后立即注销旧窗口，下次呼出立即创建新窗口。

## 非目标

- 不改变快捷提问窗口尺寸、位置策略、Provider 选择逻辑。
- 不改变“新对话”按钮当前的导航回首页行为。
- 不为主窗口 AI WebView 增加类似的隐藏后注销策略。

## 推荐方案

在 Rust 原生窗口层统一实现快捷键行为和隐藏后注销。

`quick_ask::toggle()` 是全局快捷键入口，能直接读取窗口是否存在、是否可见、是否聚焦。它应复用现有 `raise()`，该函数已经实现“临时置顶、显示并聚焦、非图钉状态下取消置顶”的逻辑。

隐藏窗口的所有入口应共用同一个隐藏函数：快捷键隐藏、顶栏“隐藏”按钮都进入该函数。这样窗口隐藏和注销策略不会因入口不同而出现行为差异。

## 快捷键行为

快捷键触发时按以下顺序处理：

1. `quick-ask` 窗口不存在：创建窗口和 AI 子 WebView，居中后调用 `raise()`。
2. 窗口存在但不可见：取消旧注销任务，显示 AI 子 WebView，调用 `raise()`。
3. 窗口存在、可见、且当前聚焦：隐藏窗口，并按重置策略安排注销。
4. 窗口存在、可见、但未聚焦：取消旧注销任务，调用 `raise()` 抬到最上方。

这里的“聚焦”必须使用窗口真实状态，而不是记录快捷键行为。用户可以手动点击窗口让快捷提问重新聚焦，实现必须尊重这个状态。

## 重置策略设置

新增前端设置字段：

```ts
export type QuickAskResetPolicy =
  | "reopen"
  | "after5m"
  | "after10m"
  | "after20m"
  | "after30m"
  | "never";
```

`Settings` 增加：

```ts
quickAskResetPolicy: QuickAskResetPolicy;
```

默认值：

```ts
quickAskResetPolicy: "after5m";
```

基础配置页新增下拉项：

- 标签：`快捷提问重置为新对话`
- 选项：
  - `重新打开时`
  - `关闭后5分钟`
  - `关闭后10分钟`
  - `关闭后20分钟`
  - `关闭后30分钟`
  - `从不`

`mergeSettings()` 对旧设置补齐默认值 `after5m`。

Rust `settings_io.rs` 读取同名字段，旧设置缺字段时也使用 `after5m`。

## 注销生命周期

隐藏快捷提问窗口时：

1. 调用 `win.hide()`。
2. 递增 `quick_ask_reset_generation`。
3. 根据当前 `quickAskResetPolicy` 处理：
   - `reopen`：立即关闭 `quick-ask` 窗口，让下次呼出重新创建。
   - `after5m` / `after10m` / `after20m` / `after30m`：启动异步定时任务。
   - `never`：不启动定时注销。

定时任务到点后，只有同时满足以下条件才关闭窗口：

- generation 仍匹配，说明期间没有新的显示、聚焦或隐藏操作让任务过期。
- `quick-ask` 窗口仍存在。
- 窗口仍不可见。
- 窗口未聚焦。

取消旧定时注销的方式是递增 `quick_ask_reset_generation`。以下场景都必须取消旧任务：

- 快捷键重新呼出隐藏窗口。
- 快捷键把已显示但未聚焦的窗口抬到最上方。
- 用户手动点击快捷提问窗口，使它重新获得焦点。
- 新创建快捷提问窗口。

因此，“关闭后 N 分钟”实际语义是：窗口持续隐藏满 N 分钟后注销。如果窗口重新显示或重新聚焦，旧定时任务必须失效，不能注销正在使用的窗口。

## 原生窗口事件

`lib.rs` 的 `on_window_event` 增加对 `quick-ask` 的 `WindowEvent::Focused(true)` 处理，调用 `quick_ask::cancel_pending_reset(app)`。

主窗口现有 close 拦截保持不变，只拦截 `main` 窗口。快捷提问窗口不拦截 close，这样注销时可以真正释放窗口和 AI 子 WebView。

## 状态结构

`AppState` 增加：

```rust
pub quick_ask_reset_generation: AtomicU64
```

该字段用于让旧定时任务失效。它不需要持久化，也不代表窗口是否显示，只是生命周期任务版本号。

## 错误处理

- 获取窗口聚焦状态失败时，按“未聚焦”处理：窗口可见时按快捷键优先 `raise()`，避免误隐藏。
- 注销任务到点后，如果窗口已不存在，直接结束。
- `win.close()` 失败时不重试，保留窗口隐藏状态；后续呼出仍可复用或再次隐藏后重新安排注销。
- 读取设置失败或字段缺失时，使用默认 `after5m`。

## 测试策略

前端 Vitest：

- `DEFAULT_SETTINGS.quickAskResetPolicy` 默认为 `after5m`。
- `mergeSettings()` 对旧设置补齐 `after5m`。
- 基础配置页显示“快捷提问重置为新对话”下拉项。
- 修改下拉项后持久化新策略。

Rust 测试：

- 覆盖重置策略到延迟时长的映射：
  - `reopen` -> 立即注销。
  - `after5m` -> 5 分钟。
  - `after10m` -> 10 分钟。
  - `after20m` -> 20 分钟。
  - `after30m` -> 30 分钟。
  - `never` -> 不注销。

手动验证：

- 快捷提问窗口隐藏时，按快捷键能呼出。
- 快捷提问窗口显示且聚焦时，按快捷键隐藏。
- 快捷提问窗口显示但被其它窗口覆盖时，按快捷键将其抬到最上方。
- `重新打开时`：隐藏后下次呼出是新窗口。
- `关闭后5分钟`：隐藏未重新打开满 5 分钟后，下次呼出是新窗口。
- 隐藏后在到期前重新呼出或手动聚焦，不会被旧定时任务注销。
